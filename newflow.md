# Nzubia Platform - New System Flow & Architecture

helper_doc_version: 1.0 

This document provides a comprehensive overview of the **Nzubia Global Backend** architecture, highlighting the recent enhancements made to support dynamic business logic and operational scalability.

## 1. System Philosophy
The platform connects **Exporters (Customers)** with **Logistics Providers (Agents)** via a managed marketplace. Detailed data verification, dynamic pricing, and secure payments are the core pillars.

## 2. Recent Key Enhancements

### A. Dynamic Commission Structure
**Before**: Commission rates were hardcoded in the codebase (fixed 10%).
**Now (New Flow)**:
- We introduced a **Global Platform Settings** module.
- Admins can change the `COMMISSION_RATE` instantly via the API (or future Admin UI) without redeploying code.
- **How it works**: When a payment is processed, the system fetches the current live rate from the database.

### B. Enhanced Data Verification
**Before**: Data collection was minimal (Email, Password).
**Now (New Flow)**:
- **Customers**: Can now register as **Businesses** with `Tax ID`, `Business Name`, and full `Address` profiles.
- **Agents**: Compulsory collection of `Insurance Certificates` (URL), `Cargo Specializations` (e.g., HAZMAT, Perishable), and `Fleet Size`.
- **Purpose**: Enables "Smart Matching" in the future by ensuring we have granular data on every user.

### C. Detailed Quoting & Shipping
**Before**: Simple "Price" field.
**Now (New Flow)**:
- **Shipments**: Explicit `Service Level` selection (Express vs Standard vs Economy) and specific needs (e.g., "White Glove Delivery").
- **Quotes**: **Full Cost Breakdown** is now structured data (Base Charge + Customs + Insurance + Platform Fee = Total). This provides total transparency to the customer.

## 3. Core Architecture Modules

### User & Authentication
- **Stack**: NestJS + Passport + JWT + Google OAuth.
- **Roles**: Distinct flows for Customer, Agent, and Admin.
- **Security**: Emails verified via OTP. Passwords hashed with bcrypt.

### Shipments & Matching
- **Entity**: `Shipment` acts as an RFQ (Request for Quote).
- **Flow**:
    1. Customer posts Shipment (Status: `REQUESTED`).
    2. Agents view filtered list (based on Origin/Dest).
    3. Agents submit Quotes (Status: `QUOTED`).
    4. Customer accepts Quote -> `BOOKED`.

### Payments & Escrow
- **Provider**: Stripe Connect.
- **Escrow Logic**: Funds are held in the Platform's Stripe account (or via PaymentIntent hold) until service fulfillment logic is triggered (future phase).
- **Split**: Platform takes `COMMISSION_RATE` %, Agent gets the rest automatically via Stripe Connect transfer.

### Platform Settings (NEW)
- **Purpose**: Hot-swappable configuration.
- **Key Settings**:
    - `COMMISSION_RATE`: Percentage taken from every transaction.
    - Future: `SUPPORT_EMAIL`, `MAINTENANCE_MODE`, etc.

## 4. Admin Workflow for Updates
To update global variables (like Commission):
1.  **Endpoint**: `PATCH /api/v1/platform-settings/COMMISSION_RATE`
2.  **Body**: `{ "value": "5.5" }` (Sets commission to 5.5%)
3.  **Effect**: Immediate for all **future** transactions.

## 5. Development & Deployment
- **Database**: PostgreSQL (TypeORM for safe migrations).
- **Storage**: Google Cloud Storage (for Insurance Certs/Docs).
- **Real-time**: Socket.io for Chat and Status Updates.
